## 1. Login Window (varA-login-window.jsx vs app/ui/login_window.py)

The current code follows **Variation A+ Refined Dark** correctly (frameless + paintEvent gradient, 3 pages, brand header, footer). The user complaint "feels disconnected" is rooted in concrete mismatches below.

### 1.1 Brand icon is the **app QIcon pixmap**, not the orange "bolt tile" defined in design 🔴
- **Design**: `varA-login-window.jsx:45-57` `LWIcon` is a 32–88 px **rounded orange tile** with linear-gradient `#FF7A45→#FF9F0A`, soft outer shadow, and white bolt SVG inside. Login page 0 in variant A uses `<LWIcon size=88>` *centered above the H1*. A+ variant keeps a 32 px LWIcon **only in the top-left brand header** + uses big typography for impact.
- **Code**: `app/ui/login_window.py:374-388` builds the brand header icon by calling `QApplication.instance().windowIcon().pixmap(32, 32)`. There is **no separate orange bolt-tile widget**, and the centered Page 0 has **no large icon at all** — only "SECURE ACCESS" + "ようこそ" text. CLAUDE memory's `feedback_brand_icon.md` says "TopBar/Login uses :/img/icon.png", but the design intent for the login screen specifically is the orange tile glyph (mockup explicitly draws it, not the app icon PNG).
- **Severity**: 🔴 — this is the single biggest "doesn't feel like the rest" cue. The window uses orange glow paint, but the focal element is a small grey app icon.
- **Fix**: build a dedicated `_LoginBrandTile(size=32)` widget that draws a `QLinearGradient(135°, #FF7A45→#FF9F0A)` rounded square (radius = size·0.27) with the bolt SVG `M13 2L4.5 13.5h6L9 22l9-12h-6.5L13 2z` rendered in white via `QSvgRenderer`. Use it both in the top-left header and consider a 56 px instance above the page-1 warn glyph.

### 1.2 Page 1 "未登録" warn glyph is a unicode `⚠`, not the SVG triangle 🟡
- **Design**: `varA-login-window.jsx:139-144` (variant A) and `:703-707` (A+) — 36 px stroke-line triangle with circle dot, on a 72×72 amber-tinted rounded rectangle (`#FF9F0A` 14% bg, 30% border).
- **Code**: `login_window.py:487-489` — `QLabel("⚠")` styled as 48 px text with `color: #FF9F0A; background: transparent`. Missing the rounded amber tile container with border.
- **Fix**: wrap glyph in a 72×72 `QFrame` with `background: rgba(255,159,10,0.14); border: 1px solid rgba(255,159,10,0.30); border-radius: 22px` — or render the SVG with `QSvgRenderer` like Google G is rendered.

### 1.3 Top-of-page "SECURE ACCESS" letter-spacing too small 🟢
- **Design**: A+ uses `letter-spacing: 0.18em` (`varA-login-window.jsx:657`) — distinctly wider than normal.
- **Code**: `login_window.py:420` sets `QFont.AbsoluteSpacing, 4` (≈4 px on 10 px font ≈ 0.4 em). That looks tighter than design intent but in *the right direction*. Design produces ≈1.8 px on 10 px text. Acceptable, but **welcomeTitle** (`#welcomeTitle`) is `font-size: 28 px / weight: 700` while design says `28 px / 800`. Bump to 800.

### 1.4 macOS traffic-light dots are clickable but only the red close one acts; design dots are decorative on Windows 🟢
- **Design**: pure decoration (`varA-login-window.jsx:33-35`). Code makes them slightly larger interactive targets (`PointingHandCursor`). Not wrong — but the yellow/green do nothing despite cursor change. Either remove `PointingHandCursor` from the inactive ones or wire them (yellow → minimize, green → no-op) for consistency.

### 1.5 Window-show animation works (slide-up 20 px + fade) but the design footer is split into two columns 🟢
- **Design**: `varA-login-window.jsx:775-786` — footer is `flex: space-between` with `© Shirokuma Power` LEFT and `jw.lee@shirokumapower.com` RIGHT, monospace, 9 px, letter-spacing 0.06 em.
- **Code**: `login_window.py:582-591` — both labels are `setAlignment(Qt.AlignCenter)` stacked vertically. Functionally fine but visually "cramped centered" instead of "anchored corners".
- **Fix**: change `_build_footer` to a `QHBoxLayout` with `addStretch()` between the two QLabels; remove `AlignCenter`.

### 1.6 Footer / header use raw hex (`#F2F4F7`, `#6B7280`, `#FF7A45`) rather than tokens 🟡
- `_apply_overrides` (`login_window.py:717-897`) hardcodes ~25 hex colors. Since the dialog is dark-only, this is functional, but it bypasses `TOKENS_DARK`. If a user requested a light login this would break. Document as design intent ("login is intentionally dark-only"), or thread `TOKENS_DARK` through.

## 2. Animations & Micro-Interactions

### 2.1 Easing curves don't match the design "spring/overshoot" curves 🟡
- **Design** (`02-components.md:415-422`): four named curves: `cubic-bezier(0.4, 0, 0.2, 1) 120ms` (hover), `cubic-bezier(0.34, 1.56, 0.64, 1) 240ms` (overshoot drawer/card-in), `cubic-bezier(0.4, 0, 0.6, 1) 180ms` (page transition), `linear 1000ms` (spinner/shimmer).
- **Code**: only `QEasingCurve.OutCubic`, `OutQuad`, `InCubic`, `OutExpo`, `Linear`, `InOutQuad` are used (grep: `app/ui/**`). The **overshoot curve** (which Qt provides as `OutBack`, ratio≈1.7) is never used — design's 240 ms card/drawer entrance has no implementation.
- **Fix**: introduce a `Curves` constants module (or extend `app/core/constants.py:Animations`) mapping each design curve to a Qt enum or a `QEasingCurve.Type.Custom` builder. Use `OutBack` (~ overshoot 1.56) for `FadeStackedWidget` page transitions and card mount.

### 2.2 Card hover lift is implemented on dashboard cards but **not** on `LeeCard` itself 🟡
- **Design** (`02-components.md:84-89`): "interactive card" → `box-shadow → md`, `transform: translateY(-1px)`, transition 120ms.
- **Code**: `app/ui/components/card.py:72-74` only changes `border-color` on hover. No shadow swap, no Y translate. Dashboard cards re-implement their own `_hover_anim` in `dashboard_cards.py:379-381` (150ms `OutQuad`) but `LeeCard(interactive=True)` consumers don't get this for free.
- **Fix**: in `LeeCard`, hook `enterEvent`/`leaveEvent` to a `QPropertyAnimation` on the `QGraphicsDropShadowEffect`'s `blurRadius`/`offset` and a `QPropertyAnimation` on the widget `pos` y-component (1 px). Match design 120 ms.

### 2.3 Skeleton shimmer is correct (1.6 s loop) but **applied in only 2 places** 🟡
- **Design** (`tokens.css:189-199`): every loading widget should display shimmer placeholders.
- **Code**: `LeeSkeleton` class exists (`app/ui/components/skeleton.py`) and is wired in `app/widgets/dashboard.py` and `google_calendar.py` only. None of the per-widget detail screens (spot, jepx, jkm, weather, hjks, briefing, gmail) display a skeleton during their first fetch — they show empty space or a blank table.
- **Fix**: each detail page should overlay a `LeeSkeleton(kind="card")` or stack of `LeeSkeleton(kind="line")` until first `fetch.success` signal.

### 2.4 Page transition uses `FadeStackedWidget` (200 ms `InOutQuad`) — design says 180 ms `cubic-bezier(0.4, 0, 0.6, 1)` 🟢
- `app/ui/common.py:118-119`. Off by 20 ms and slightly different curve — visible only side-by-side. Low priority.

### 2.5 The `calPop` keyframe (date-picker popup scale-in) is missing 🟢
- **Design** (`varA-datepicker.jsx:90-91, 192`): `animation: calPop 0.15s ease-out` from `opacity: 0 / translateY(-4px) scale(0.97)`.
- **Code**: `LeeMiniCalendar` renders instantly (no animation hook). Add a `QGraphicsOpacityEffect` + `QPropertyAnimation` triggered on `showEvent`.

## 5. Pivot Table / Chart Frame styling (varA-pivot.jsx)

### 5.1 LeePivotTable cell-bg uses `setBackground(QBrush(QColor))` correctly but loses the `0.18 alpha` because Qt model `BackgroundRole` paints over the row stripe 🟡
- **Design** (`varA-pivot.jsx:37-41`): cell `background: rgba(52,168,83,0.18)` with foreground `#0b8043`.
- **Code**: `charts.py:382-384` does `item.setBackground(QBrush(bg_qc))` with rgba alpha properly converted by `_rgba_to_qcolor`. **However** `QTableWidgetItem` background in many Qt styles applies *behind* the alternating row pattern, so the alpha looks darker than design.
- **Fix**: disable `setAlternatingRowColors(False)` (default) and verify `QTableView::item { background: transparent; }` so model background fully shows. Test with reserve mode (lowest sample range).

### 5.2 Pivot **row hover** highlight not implemented per design 🟡

- **Code**: `charts.py:491-493` only has `QTableWidget::item:hover { background: bg_surface_3; }` — single cell, no row/col axis highlight.
- **Fix**: install eventFilter on the table viewport, track mouseMove → set `setCurrentCell(row, col)` with a custom selection-color or a transparent overlay row/col rect.

### 5.3 LeeChartFrame (defined in `detail.py`) — verify accent gradient bar 🟢
- Design has chart frames with a 4 px top/left accent bar matching widget color. Confirm `LeeChartFrame` does this — exists per `__init__.py:30`.

### 5.4 LeeBigChart hover dot uses **white fill** regardless of theme 🟢
- `charts.py:124-127`: `brush=pg.mkBrush(255, 255, 255, 230)`. Design (`varA-detail-atoms.jsx`) shows a *theme-tinted* dot with accent border — fine, but on light theme the white blends in. Use `bg_surface` color instead.

## 9. Reactivity / responsive behavior

### 9.1 Dashboard responsive breakpoint is hardcoded at `_RESPONSIVE_BP_PX = 900` 🟢
- `dashboard.py:67`. Design specs don't precisely require this number; acceptable.

### 9.2 Sidebar collapse persistence works ✅
- `MainWindow._save_group_states / _restore_group_states` (`main_window.py:422-433`) properly persists per-group collapsed state to settings under `sidebar_collapsed_v2`.

### 9.3 TopBar tab → dashboard auto-navigate is implemented but blank `tab_key` returns early 🟢
- `main_window.py:364-377` — when filter is cleared (tab_key=""), no navigation happens. Design intent ambiguous; current behavior preserves current page. OK.

### 9.4 No theme switch transition for child detail screens — only overlay fade on the **window** 🟡
- `main_window.py:497-530` `_toggle_theme` grabs the current pixmap, swaps theme, fades the overlay. Detail widgets that have **internal `apply_theme`** are called via `_sync_theme()` *before* the fade, so during the 250 ms fade the new theme is already partially visible behind the overlay — works but is jarring for charts that re-layout.
- **Fix**: defer per-widget `set_theme` calls to **after** overlay fade completes (move call into the `finished` lambda).

### 9.5 Window-resize (`mouseMoveEvent` in main_window.py) intercepts every child's mouse move — heavy 🟢
- `main_window.py:243` installs `QApplication.installEventFilter(self)`. Necessary for frameless edge-resize but means every mouse event traverses the filter. Mostly fine, micro-optimization opportunity only.
